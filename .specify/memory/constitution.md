## 核心原則

### I. 規格驅動

所有功能必須從清晰的自然語言需求開始。規格是唯一的需求真實來源。MUST：
- 每個功能必須有明確的 spec.md（由 speckit.specify 生成）
- 規格必須包含清晰的目標、範圍和驗收標準
- 在設計或實中前必須獲得利益相關方批准
- 規格是可追蹤性和問責制的基礎

### II. 設計優先

實現必須基於詳盡的設計文件。MUST：
- plan.md 必須存在於實現前，定義技術方案和架構決策
- 設計文件必須包含替代方案分析和權衡
- 複雜功能需要詳細的 API 或工作流圖
- 設計變更必須更新並重新批准規格

### III. 自動化工作流

從需求到任務的轉換必須自動化。MUST：
- 使用 speckit 工具鏈生成規格、計劃和任務
- 不手動修改 speckit 生成的結構化文件
- 任務必須有依賴關係排序（由 speckit.tasks 生成）
- 工作流的每個階段必須有檢查點和驗證

### IV. 測試驅動開發 (TDD)

測試優先是強制性的。MUST：
- 測試在實現前完成（Red → Green → Refactor）
- 所有公開 API / 功能必須有單元測試
- 整合測試覆蓋跨元件互動
- 測試失敗不得忽視；必須修復或更新規格

### V. 簡潔性優先

避免過度設計。MUST：
- YAGNI 原則：不要設計未來的需求
- 每個特性應以最小的複雜性完成
- 規格必須簡潔明確，避免不必要的技術細節
- 交易簡潔性可以被清晰的業務需求或性能要求正當化

## 開發標準

所有開發必須遵循以下標準：

**版本控制**：
- 每個主要工作流階段（specify → plan → tasks → implement）都必須進行 git commit
- Commit 信息必須清晰指明是哪個工作流階段和功能
- 不允許跳過規格或計劃階段

**文件完整性**：
- 原始規格檔案（spec.md, plan.md, tasks.md）必須受版本控制保護
- 不得在框架模板應用期間覆蓋或刪除規格文件
- 所有配置和模板更新必須驗證不影響現有規格

**工具鏈遵從**：
- 所有規格相關文件必須使用 speckit 定義的模板
- 模板符合本規格憲法
- 定制必須與核心原則對齊

## 品質保證

**規格驗證**：
- speckit.clarify 用於澄清不明確的需求
- speckit.analyze 必須驗證 spec.md、plan.md 和 tasks.md 的一致性

**工作流檢查**：
- 實現前必須運行完整分析
- 任務完成必須更新 tasks.md 並提交 git

**GitHub 整合**：
- Tasks 可轉換為 GitHub issues（使用 speckit.taskstoissues）
- 每個 issue 必須可追蹤回原始規格

## 治理

**憲法權威性**：
本憲法超越所有其他實踐和指導。遵守規格驅動的工作流是非協商的。

**修訂流程**：
- 憲法修訂必須記錄變更原因
- 版本必須根據語義化版本管理：
  - **MAJOR**：原則移除或根本性重新定義
  - **MINOR**：原則或章節補充或重大擴展
  - **PATCH**：澄清、措詞改進、錯誤修正
- 修訂後必須更新所有依賴模板（plan, spec, tasks）

**符合性審查**：
- 所有新功能必須聲明對各原則的符合性
- 異常必須記錄並追蹤為 TODO 或規格例外
- 審查者負責驗證工作流完整性
